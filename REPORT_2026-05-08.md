### 1-2. 코드상 마이그레이션 완료 사항

- **LLM 호출 경로**: `google.genai` 제거 → `openai` 클라이언트가 `http://localhost:8080/v1` 로 호출 ([generator.py:57](core/generator.py#L57)). `temperature=0.6 / top_p=0.9 / seed=42 / max_tokens=16384` 로 고정.
- **think 모드 차단**: 시스템 프롬프트 끝에 `/no_think`, `extra_body={"chat_template_kwargs": {"enable_thinking": False}}`, `<think>...</think>` 블록 사후 제거 정규식 ([generator.py:18, 80, 94, 103](core/generator.py)).
- **재시도 로직**: `tenacity` 로 `APIConnectionError`·`APIStatusError` 5회 지수 백오프 (4→60s).
- **서버 헬스체크**: `main.py` 진입 시 `/v1/models` 호출, 실패 시 사용자에게 mlx_lm.server 기동 명령어 안내 ([main.py:40-49](main.py#L40-L49)).
- **이미지 백엔드 교체**: `requests.post(HF_API_URL)` → `subprocess.run(mflux-generate ...)` ([image_generator.py:38-100](core/image_generator.py#L38-L100)). 기본 모델 `FLUX.1-dev` / 50-step / guidance 3.5 / 1024². negative 프롬프트 자동 분리, `MFLUX_*` 환경변수로 오버라이드 가능.
- **`requirements.txt`** 정비: `google-genai` 제거, `openai` + `mflux` 도입.
- **설정**: `LLM_BASE_URL` / `LLM_MODEL` 환경변수 추가, 기본값 `mlx-community/Qwen3-32B-4bit` ([settings.py:25-26](config/settings.py#L25-L26)).

## 2. 실측 결과 (테스트 로그 기반)

### 2-1. 본문 생성 — `mlx-community/Qwen3-32B-4bit` (5월 6~7일 4회)

| 실행 | 자막 길이 | 생성 시간 | 제목 파싱 | 이미지 프롬프트 |
|---|---|---|---|---|
| 2026-05-06 16:31 | 10,934자 | **316s** | ✅ 정상 | ❌ fallback (generic) |
| 2026-05-06 17:49 | 10,934자 | **245s** | ❌ `[제목: "..."]` 그대로 포함 | ❌ fallback (generic) |
| 2026-05-07 09:09 | 10,934자 | **307s** | ❌ `TITLE: "..."` 그대로 포함 | ❌ fallback (generic) |
| 2026-05-07 09:29 | 10,934자 | **305s** | ✅ 정상 | ✅ 정상 (영문 다단락) |

**관찰:**
- **속도**: M4 Max 128GB · Qwen3-32B-4bit 기준 입력 ~11K 자막 · 출력 수천 토큰에 **4~5분/편**. 클라우드(Gemini Flash 27~32s) 대비 ~10배 느리지만 발행 빈도가 일 1~2편이면 실용 가능.
- **출력 포맷 안정성 문제**: 4회 중 **제목 마커 누수 2회 / 이미지 프롬프트 마커 누수 3회**. 즉 50% 미만의 안정성. Gemini 시절에는 `_IMAGE_MARKER_LINE_RE` 가 평문 `IMAGE_PROMPT:` 단일 라인을 가정해도 거의 항상 들어맞았으나, **Qwen 계열은 헤딩(`## IMAGE_PROMPT`)이나 굵게(`**IMAGE_PROMPT:**`) 또는 `[제목: "..."]` 같은 변형을 출력하는 빈도가 높음**.
- 정규식은 `^\s*\*{0,2}#{0,3}\s*IMAGE_PROMPT\s*:?\s*\*{0,2}\s*$` 로 헤딩·굵게는 잡도록 이미 보강되어 있음 ([generator.py:21-24](core/generator.py#L21-L24)). 그럼에도 fallback 이 발생한다는 건 모델이 **마커 자체를 출력하지 않거나, 본문 중간에 누수시키거나, 다른 문구로 치환**하는 케이스가 있다는 뜻.

- 첫 실행 시 HF 라이선스 수락 + 가중치 ~24GB 다운로드 1회 필요.

## 3. 문제 정리 (우선순위 순)

### P0 — 본문 출력 파싱 안정성
- **증상**: 마커 누수 → 제목에 `TITLE:` 포함, 이미지 프롬프트가 fallback generic 으로 떨어짐. 4회 중 절반이 영향.
- **현재 방어선**: think-block 제거, 헤딩/굵게 마커 정규식 보강, fallback 제목·프롬프트 자동 채움.
- **부족한 부분**: 모델이 마커를 아예 빠뜨리거나 다른 키워드로 바꾸는 경우. 이는 정규식이 아니라 **프롬프트 + few-shot 예시 + 출력 검증 루프**로만 해결 가능.

### P1 — 이미지 생성 속도 (현 12분/장 → 10초대 목표)
- 현재 `FLUX.1-dev BF16 50-step`: 품질 최상이지만 발행 1편당 12분은 자동화 가치가 떨어짐.
- RESEARCH 1순위 `FLUX.1-schnell-mflux-8bit` (4-step, 검증된 M4 Max 128GB 측정값 ~10.5s) 로 전환 시 **속도 ~70배 / 메모리 ½**. 품질 차이는 블로그 히어로 이미지 용도에 충분.
- 환경변수만 바꾸면 됨: `MFLUX_MODEL=dhairyashil/FLUX.1-schnell-mflux-8bit`, `MFLUX_BASE_MODEL=schnell`, `MFLUX_STEPS=4`, `MFLUX_GUIDANCE=0`.

### P2 — LLM 모델 업그레이드 (Qwen3 → Qwen3.5)
- RESEARCH 가 권하는 1순위 `Qwen3.5-32B-4bit` 로의 교체. 한국어·영어 모두 한 단계 향상되었고 본 사용자가 이미 별도 벤치마크 프로젝트에서 검증함. P0 의 마커 누수 빈도를 같이 끌어내릴 가능성도 있음 (지시 준수도 향상).

### P3 — 자막 의존
- yt-dlp 자막이 없는 영상에서 즉시 `RuntimeError`. 현재 4회 테스트는 모두 자막 있는 영상이라 가려져 있을 뿐.

### P4 — 본문 생성 4~5분 관찰성
- mlx_lm.server 가 별도 터미널에서 도는 구조라 **TPS·thinking 토큰·메모리** 가시성이 없음. 회귀 측정·튜닝에 불편.

## 4. 향후 계획 / 자동화 테스트 권고안

### A. 회귀 테스트 스위트 도입 (P0 직격, 1~2일)

목표: "로컬 LLM 모델·프롬프트·파라미터를 바꿀 때마다 동일 입력에 대해 출력 품질이 무너지지 않았는지를 1분 안에 자동 판정."

**구성안:**

```
scripts/regression_test.py
├─ tests/golden/           # URL 5~10개 + 기대 메타
│   ├─ ev_market.json      # url, topic_keywords, type=뉴스
│   ├─ lfp_compare.json    # url, topic_keywords, type=비교
│   └─ ...
└─ tests/fixtures/         # 자막을 미리 캐시 (yt-dlp 호출 안 함, 재현성 ↑)
    └─ <video_id>.txt
```

**실행 시 측정·기록할 메트릭:**

| 메트릭 | 통과 기준 | 측정 방식 |
|---|---|---|
| 제목 파싱 성공 | 100% | `TITLE:`/`[제목` 누수 정규식 검사 |
| 이미지 프롬프트 비-fallback 비율 | ≥90% | generator 가 fallback 분기를 탔는지 플래그 추가 후 집계 |
| 본문 길이 | 1,000~3,500자 | `len(markdown)` |
| H2 개수 | 2~4개 | `\n## ` 카운트 |
| 표기 통일 위반 | 0 | "리튬 이온" / "리튬-이온" / "LFP배터리(공백 누락)" 정규식 |
| FAQ 헤딩 표기 | `## 자주 묻는 질문` 만 사용 | "FAQ" / "Frequently" 등장 검사 |
| 본문 내 전화번호 누수 | 0 | `031-` 정규식 |
| 생성 시간 | 회기 baseline ±20% | LLM 호출 wall time |

**통과/실패는 단순 boolean이 아니라 점수**로 — 모델·파라미터 변경 후 점수가 개선되는지 보면 충분.

WordPress 발행은 **dry-run 모드**(아래 B) 위에서 실행 → 실제 사이트에 쓰레기가 안 쌓임.

### B. Dry-run 모드 (ROADMAP 9번, 2~3시간)

`python main.py <url> --dry-run` → WordPress 호출·이미지 생성을 **모두 스킵**, `output/<slug>.html` 과 `output/<slug>.image_prompt.txt` 만 로컬 저장. 회귀 테스트가 의존하는 전제. 이미지 스킵 옵션 `--no-image` 도 함께.

### C. 이미지 모델 환경변수화 + AB 비교 자동화 (반나절)

이미 [scripts/test_image_models.py](scripts/test_image_models.py) 골격이 있음. 다음 작업만 추가:
1. `MFLUX_MODEL` 등 환경변수를 도입한 김에 `.env.example` 갱신.
2. test_image_models 의 결과 폴더에 **세로 4컷 비교 그리드 PNG** 자동 생성 (Pillow). 사람이 평가하기 쉬움.
3. 평가 결과를 `data/image_model_pick.md` 에 메모 → `image_generator.py` 기본값 변경.

**1차 비교 대상:** mflux-schnell-8bit (속도 후보) · mflux-dev BF16 (현재 품질 베이스라인) · mflux-schnell BF16 (중간) — 같은 프롬프트·시드 4회로 일관성도 같이 확인.

### D. 자막 비의존 모드로의 확장 (사용자 지정 방향)

ROADMAP 에는 "STT fallback" 단일 항목만 있음. 본 보고서에서 두 갈래로 분리해 우선순위 명시:

#### D-1. STT fallback (자막은 있지만 보강이 필요할 때 / 자막이 없을 때)
- `mlx-whisper large-v3-turbo` (~6GB) 도입.
- `transcriber.py` 인터페이스를 **자막 우선 → 실패 시 STT** 순으로 변경.
- 첫 도입 비용: 모델 다운로드 1회 + 함수 1개. M4 Max 128GB 동시 상주 (LLM 18 + Image 12 + STT 6 = 36GB) 여유 충분.

#### D-2. 키워드 기반 생성 (YouTube 비의존, **사용자가 디벨롭 방향으로 지정**)
현재 `data/keywords_trending.md` 가 매주 자동 갱신되고 있음. 이걸 **소스로** 쓰는 새 엔트리포인트.

```
publish_from_keyword.py "리튬인산철 배터리 안전성" --type 가이드
publish_from_keyword.py --auto       # 트렌드 상위 3개 자동 선택
```

**설계 단계의 핵심 질문:**
- **사실 근거**: 자막이 없으면 모델이 환각으로 사실을 만들 위험이 가장 큼. 후보 (a) 시드 + 자체 회사 데이터(`data/`)만 허용하고 사실 확장 금지, (b) 검색 결과 N건을 컨텍스트로 주입(웹 검색 도구 추가), (c) 회사 자체 자료실(제품 스펙·매뉴얼) 인덱싱하여 RAG. 본 단계에서는 **(a)** 로 시작 → 결과 보고 (b) (c) 단계적 도입.
- **주제 선정**: 트렌드 상승률이 높은 키워드 자동 선택 vs 운영자 수동. **수동 우선** (안정적 운영 먼저, 자동은 충분히 안 굳은 단계에서 위험).
- **중복 회피**: ROADMAP 10번 (`processed.jsonl`) 과 통합. 트렌드 키워드도 발행 이력에 남기고 N일 내 동일 키워드 재발행 금지.

**최소 구현 (1~2일):**
1. `core/generator.py` 의 `generate_blog_post` 시그니처에서 `transcript_text` 를 옵셔널로 + 새 파라미터 `seed_topic: str` 추가.
2. 시스템 프롬프트에 분기: "자막이 있으면 자막 기반·없으면 시드 토픽 기반, 둘 다 사실 창작 금지 원칙은 동일."
3. 새 엔트리포인트 `publish_from_keyword.py`. 자막 추출 단계 스킵, 나머지 트렌드·이미지·발행은 동일.

### E. 발행 후 결과 측정 — ROADMAP 16번 선행 추천

로컬화로 비용 구조가 바뀌었음(클라우드 토큰비 → 자기 시간·전기). 그렇다면 **무엇을 발행해야 ROI가 좋은가**를 데이터로 알아야 함. ROADMAP 16번(GA4 + Search Console + AEO 마크업)을 우선순위로 끌어올리길 권장. 데이터 누적 4~8주 필요하므로 **지금 깔아둬야 D-2 가 의미 있는 시점에 데이터가 있음**.

## 6. 참고 / 보안 메모

- `.env` 에 평문 자격증명(WordPress 비밀번호, API 키)이 들어 있음. `.gitignore` 처리는 되어 있으나, 향후 협업·저장소 공개 시점이 오면 `.env.example` + 실제 `.env` 분리 + 자격증명 만료·교체 절차 필요.
- 로컬화로 외부 API 호출은 줄었으나, **HF 가중치 첫 1회 다운로드**(LLM 18GB + 이미지 12~24GB + STT 6GB)는 네트워크와 디스크에 부담. 도입 순서를 schnell-8bit → STT-turbo 순으로 잡으면 실패 비용을 낮출 수 있음.
- 본 보고서 §2의 데이터는 `logs/automation.log` 와 `logs/generation_*.md`, `logs/image_test_*/report.md` 만을 출처로 함. 추가 측정은 §4-A 회귀 테스트 도입 후 자동 누적되는 구조로.
